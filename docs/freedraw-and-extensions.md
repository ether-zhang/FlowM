# 手绘理解 与 后处理扩展（pass vs op）

> 配合 [architecture.md](architecture.md)（三层 + LayoutPass 管线）。本文收尾「手绘(freedraw)理解」这条线,并给出**新能力该做成 pass 还是 op** 的判据 + 一批**设计已清、按需再建**的扩展。

## 1. 出发点:手绘是用户的输入

手绘 = 用户画给模型**读**的;模型**至多修改、基本不生成、几乎不移动**它。实测暴露两类失败:
1. **读不准**:没认出两个矩阵是 M×N / N×M(角上的小手写标注)。
2. **放不准**:补的结果框被甩到空地、丢了"行对齐 A、列对齐 B"。

根因:手绘在数据里**语义不透明**——每条 `draw` 是按抬笔切的笔画,bbox 真实但**无语义**;一张图几十条笔画,既**淹没列表**又**原子化**了本该读成"一个图形"的格式塔;徽标还**压在笔画上**遮字。

## 2. 已落地(本期)

| 改动 | 作用 | 提交 |
|---|---|---|
| 折叠手绘 + 不给笔画橙标 | 列表 18 行 → 1 行;去遮挡、去原子化 | `208f47f` |
| 复核图保留手绘 + 徽标移到形状上方 | 复核能看到"手绘+新work"拼合;小字不被盖 | `b8e4488` |
| **邻近聚类成 region + 蓝标 `[Bn]`** | 每个手绘图形一个蓝色句柄(`clusterDrawRegions`),文图同套确定性聚类;模型能**整片引用** | `5c0d0a0` |

效果:读图准确度**大幅提升**(M×N/N×M 读对),放置也随之变好(模型读懂就自己放对了)。坐标/ id **框架内部仍保留**(只是不进 prompt),为将来"整片操作"留好。

## 3. 关键判据:Pass 还是 Op

加新能力时,先分清它属于哪类——错配会把不属于 pass 的塞进 pass。

- **Pass(后处理)**:框架**自己算出几何/样式**,去**实现一个被声明的结构**或**一条自动规则**;模型**不提供目标值**,是**派生**的。受门控/scope 约束(意图 B);或无歧义恒跑(不变式 A)。
  例:`normalizeSpacing` 从"这是 flow"算出列对齐——没有"移到哪"的输入。
- **Op**:模型**显式给出目标/值**的**直接操作**。
  例:`move_shape(id,x,y)`、(未来)`set_color(id,蓝)`。

> 一句话:**有"目标坐标/值"要模型给 → Op;只有"关系/规则"要框架算 → Pass。**

## 4. 暂缓的扩展(设计已清,按需再建)

| 想做的事 | 归类 | 它会做什么 | 触发条件(出现才建) | 为何暂缓 |
|---|---|---|---|---|
| 移动整片 region | **Op** | 扩展 `move_shape` 收 `[Bn]` → 平移成员笔画(只 translate、不重塑) | 出现"让模型挪/重排我的手绘"的真实用例 | 当前模型基本不移动用户手绘,无需求 |
| 把新形状锚定到 region 旁(/任意形状作锚点) | **Pass**(align/place 实现器) | 从"C 置于 [B1] 右、列对齐"这类**相对关系**算出 C 的位置,锚点冻结只动新形状 | 模型自放置又开始失败(像最初矩阵 C 那样) | 2a 读图变准后**模型自己放对了**,大概率**冗余** |
| 上色 | **Op** 或 **Pass** | 模型命令"涂这个"=Op;框架**按角色自动**配色(判断节点/起止/判断)=Pass | 想要"一致配色"的产品诉求 | 与手绘线无关;自动配色是**最像样的未来 pass**,但无当下需求 |

**可插拔的 `LayoutPass` 设计意味着这些以后任何时候都能无痛加入**——所以现在**更没必要预建**。等某个具体用例落地,就**精准建那一个**(对应的 op 或 pass),而不是一把铺开。这与 `align`/`grid`/`contain` 实现器**同一条 YAGNI**:框架不投机替模型做它自己已经能做好的事。

## 5. 收尾

手绘这条线在 **2a 收工**:读 + 引用(`[Bn]` 句柄)是高价值且够用的部分。"操作 region / 上色"按上表分类、按需再建。判据(§3)是这套后处理体系**长期的扩展守则**。
