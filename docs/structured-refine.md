# 结构化 Refine 架构（草案）

> 分支 `feat/structured-refine` 的设计基准。目标：把"后处理"从**盲目全跑**改成
> **模型声明结构、框架照着精确实现**，让流程图 / 自由图 / 手绘在**同一张画布**共存而互不误伤。
> 这是接口草案，等砍。

## 1. 核心拆分：A（不变量，盲跑）vs B（意图敏感，门控）

判据一句话：**这步会不会"移动 / 改大小"一个节点？**

- **不会**（只决定箭头怎么搭到、绕过已摆好的节点）→ **A，盲跑**
- **会**（挪节点 / 改节点尺寸）→ **B，门控**

**为什么 A 无条件**：模型已定了拓扑（谁连谁）+ 位置（节点在哪）。二者定死后，
箭头接触点是**被推导的唯一值**——没人会要一条飘着、会跳的箭头。意图无分歧空间，直接算。
它不是"调整布局"，是**补全模型算不准的派生几何**。

**为什么 B 要门控**：节点的位置/尺寸**本身就是意图**。同一画面（两框重叠）正确答案随意图而变
（故意嵌套 → 保留；手误叠上 → 推开）。框架看像素分不出 → 必须问模型。

| pass / 操作 | 动节点？ | 类 | 无声明时的默认 |
|---|---|---|---|
| 绑定端点求解 / 不动点 | 否 | **A** | 永远跑 |
| 箭头路由（绕障/弓） | 否（只动箭头） | **A** | 永远跑 |
| 多端口 focus 分配 | 否（只动箭头搭点） | **A** | 永远跑（启发式收尾） |
| `avoidPass` 消重叠 | 是 | **B** | **冻结**（不动） |
| `spacingPass` flow 匀缝 | 是 | **B** | **冻结** |
| label 撑开尺寸 | 是（改尺寸） | **B** | 默认开，被固定尺寸结构覆盖 |

## 2. 管线：refine 插在【首图】与 B 之间

```
模型出 ops
  → 跑 A（不变量，盲跑；首图里箭头就已贴边）
  → 渲染【首图】
  → 模型看首图（+ 用户请求 + 领域知识），吐两样：
       ① 纠错 ops      —— "这俩我摆重叠了，挪一下"（模型自己改坐标）
       ② 结构声明      —— 见 §3，挂在具体 id 上，反映用户语境下"本该长什么样"
  → 只在【被结构声明覆盖 且 属于 B 的合法职责】处跑 B
  → 跑 A（节点动过了，箭头重新贴边）
  → 渲染【终图】
```

- A 跑两次（B 前后各一次），因为 B 挪了节点、箭头要重贴。
- **两把钥匙**：一处 B 动作发生 = ①被结构声明授权 **且** ②是后处理本分（消重叠/匀缝/路由/尺寸）。
  模型不能借此让框架做语义重排；框架也不能越过声明自作主张。
- **条件门控省往返**：首图里若没有任何 B 候选（无重叠、无 flow 链、无结构声明），**跳过这次模型往返**，只交 A。

## 3. 结构声明词汇（最小集，靠实测长）

模型输出的是一组**挂在 id 上的结构关系**，每条有作用域（一批 id）。MVP 先上能覆盖
流程图 + 常见自由图的几个；**没建模的结构 fall through 成"冻结"**（见 §4），永不弄坏。

| 关系 | 语义 | 作用域 | 实现器（B） | 与现有 pass 映射 |
|---|---|---|---|---|
| `flow` | 自上而下（或左右）流链 + 分支侧 | 主干节点序列 | 拓扑匀缝 + 近轴吸附 | `normalizeSpacing`（已有） |
| `nonOverlap` | 这组互不重叠 | 一组 id | 最小位移推开 | `resolveOverlaps`（已有） |
| `contain` | A 包含 B,C…（嵌套） | 父 id + 子 id | 子在父内、推开时不越父界；父按子长大 | 新增 |
| `grid` | N×M 等尺寸等距矩阵 | cell id 列表 | 算精确格点坐标 | 新增（取代模型盲吐坐标） |
| `align` | 顶/底/中 对齐成行 / 列 | 一组 id | 投影到公共轴 | 新增（`normalizeSpacing` 近轴吸附的泛化） |
| `freeze` | 原样别动（手绘/草图） | 一组 id | 空操作（显式标记） | 默认行为，可显式声明 |

设计纪律：
1. **词汇小且可组合**；靠实测加，不一次铺满。
2. **冲突要有优先级**（如 `contain` vs `nonOverlap` 把子推出父 → `contain` 优先）。
3. **作用域尽量不重叠**或显式分层，免得两个实现器互拉（参考 port-focus 与绕障曾打架）。

## 4. "不落任何情况" = 默认冻结（逃生门，非错误态）

一组形状若**不被任何结构关系覆盖**：
- 只跑 **A 不变量**（箭头照样贴边）；
- 所有空间重排 **一律不动**，按模型摆的留着。

由此得到一个很强的性质：**词汇可以不完整，系统仍然安全**——没建模的结构自动退回
"模型原图"，永远不会"被弄坏"。推论：
- **不需要一次性穷举所有关系**；先只上 `flow` + 箭头不变量就是自洽 MVP；
- 结构声明是**模型按区域 opt-in 主动要帮助**，不是强制全量分类；
- 漏声明可在首图复核补上；模型判定"无结构"时，冻结就是对的。

## 5. 依赖：视觉锚点（读侧）是写侧门控的前置

§2 的"结构声明挂在具体 id 上"要求模型能**精确指认是哪几个形状**（"保 #A⊃#B、消 #C/#D"）。
这正是读侧 **set-of-mark 视觉锚点**（导出带 id 标注的图）要解决的 image↔id 对应。
**所以读和写在这里不解耦：锚点是门控的地基。**

## 6. 落地分阶段

1. **A/B 标注重构**（本分支第一刀，零行为变化）：给 pass 打 `kind: invariant|intent`，
   把 `arrowPass` 归 A、`spacing/avoid` 归 B；编排上预留"先 A → (gate) → B → A"。
2. **视觉锚点**（读侧）：导出带 id 标注，模型能指认形状。
3. **结构声明 schema**（§3 最小集）：模型怎么声明、挂哪些 id、如何回传。
4. **B 改为按作用域跑** + 首图→声明→B→终图 的门控（含条件门控省往返）。
5. 补 `contain` / `grid` / `align` 实现器，靠实测扩词汇。
